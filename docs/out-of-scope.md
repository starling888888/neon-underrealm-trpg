# 初期スコープ外の開発項目

このドキュメントは、ネオン・アンダーレルムTRPG ルールサイト初期実装において、Codex 等が余計な機能を実装しないように、明示的に初期スコープ外とする開発項目を定義する。

初期実装の目的は、PL向けの基本ルール、世界観、キャラクター作成、Webキャラクターシート、データ、アイテム、戦闘ルール、成長ルール、更新履歴を、静的サイトとして安定して公開・更新できる状態にすることである。

1st stepの目的は、ルールサイト公開の初回告知を見た人がなるべく長くサイトを読み、「遊んでみたい」と思えることとする。完了は、初回告知時点でPLが遊ぶ判断と参加準備をするために十分なコンテンツが公開されている状態と定義する。

実装順序、ブランチ分割、作業粒度は `docs/plan.md` を優先する。

要件定義は `docs/requirements.md` を参照する。

本ドキュメントと `docs/plan.md` / `docs/requirements.md` の間に齟齬がある場合、初期開発の作業計画・スコープ判断については `docs/plan.md` を優先し、必要に応じて本ドキュメントを追従修正する。

---

## 1. GMガイド

GM向けの詳細な運用ガイド、シナリオ作成支援、エネミー運用指針、GM裁定ガイドは初期スコープ外とする。

初期公開では、PLがゲームを理解し、キャラクター作成・基本ルール・戦闘ルール・データ参照を行えることを優先する。

将来的に `/gm` 配下へ追加できる構造は許容するが、初期実装ではページ・ナビゲーション・未完成導線を作成しない。

---

## 2. シナリオ本文

公式シナリオ、サンプルシナリオ、キャンペーン本文、ハンドアウト本文の公開機能は初期スコープ外とする。

初期公開では、ルールサイトとしての基盤を優先し、シナリオ管理・シナリオ一覧・個別シナリオページは実装しない。

将来的に `/scenarios` 配下へ追加できる余地は残すが、初期実装では空ページや準備中ページを作成しない。

ただし、`/rules/scenario-play` はシナリオ本文を掲載するページではなく、PL向けのシナリオ進行ルールを説明するページとして扱うため、初期スコープ内とする。

---

## 3. キャンペーン管理機能

キャンペーン進行、セッション履歴、PC成長履歴、覚悟累積、信用管理などをWeb上で管理する機能は初期スコープ外とする。

本サイトは、初期段階ではルールとデータを公開・参照するための静的サイトであり、プレイログやキャンペーン状態を管理するアプリケーションではない。

DB、ログイン、保存機能を必要とする実装は行わない。

---

## 4. エネミー・NPC・GM専用データ

エネミー、GM専用データ、シナリオ用データは初期スコープ外とする。NPCについては、`/world` の世界観本文に含める紹介だけを初期スコープ内とする。

初期対象データは、PL向けに必要なスキル、アイテム、流儀、生き様、リリースノートを基本とする。

NPC紹介は、`/world`の世界観本文に限り、生成JSONを参照する`NpcCard`で表示する。個別画像と、個別画像がない場合の`no_image.webp` fallbackは初期スコープ内とする。NPC一覧ページ、詳細ページ、検索、フィルタ、並べ替え、ページネーション、カードからの詳細遷移、エネミー・GM専用データは初期スコープ外とする。エネミー、NPC、GM専用データを将来的に追加できる構造は妨げない。

---

## 5. キャラクター作成ウィザード

質問に答えてキャラクターを作成するウィザード、能力値自動計算、スキル選択支援、アイテム選択支援は初期スコープ外とする。

初期公開では、キャラクターメイキング手順の説明とデータ参照までを対象とする。

キャラクター作成を補助するインタラクティブ機能は、ルールサイト本体が安定した後の拡張候補とする。

---

## 6. Webキャラクターシートの初期範囲外

ブラウザ上でキャラクターを作成・編集し、遊ぶために必要な情報を確認できるWebキャラクターシートは、`docs/plan.md` の `ex-02-web-character-sheet` として初期スコープ内とする。

ログイン、ユーザー端末に依存しないサーバー・DB・クラウドへのキャラクターデータ保存、URL共有、PDF出力、Google Sheets連携は初期スコープ外とする。

入力項目、作成手順、ユーザー端末内の作業継続用保存・復元を含むブラウザ内状態の扱いは、`ex-02-web-character-sheet` で定義する。

---

## 7. ダイスローラー

Web上で判定を行うダイスローラー、達成値・効果値・気合獲得の自動計算機能は初期スコープ外とする。

ルール本文内で判定方法を説明することは行うが、実際にダイスを振るインタラクティブ機能は実装しない。

期待値計算ツール、成功率計算ツール、判定補助ツールも初期スコープ外とする。

---

## 8. 戦闘シミュレーター・戦闘処理支援ツール

攻撃、リアクション、コンボ、ダメージ計算、カウンター、カバーリングなどをWeb上で処理する戦闘シミュレーターは初期スコープ外とする。

戦闘処理支援ツール、コンボビルダー、ダメージ計算機、行動順管理、状態管理なども初期スコープ外とする。

初期公開では、戦闘ルールを読める形で整理し、必要なデータへ参照できることを優先する。

ルール処理の自動化は、ルール本文・データ構造・表示形式が安定してから検討する。

---

## 9. データ編集用CMS

ブラウザ上でスキル、アイテム、リリースノート、本文を編集するCMS機能は初期スコープ外とする。

初期運用では、本文はMarkdown / MDX / Astroページ、表形式データはローカルExcel、公開用データは変換済みJSONで管理する。

管理画面、認証、DB、投稿フォーム、編集UIは実装しない。

---

## 10. ログイン・認証機能

ユーザー登録、ログイン、権限管理、管理者画面、編集者権限などの認証機能は初期スコープ外とする。

本サイトは無料公開の静的サイトとして構築し、閲覧者ごとの状態を持たない。

認証を必要とする機能は初期実装に含めない。

---

## 11. コメント・投稿機能

ユーザーコメント、フィードバック投稿、掲示板、レビュー投稿、ルール質問投稿などの投稿機能は初期スコープ外とする。

外部サービスへのリンクを設置することは許容するが、サイト内に投稿データを保存・表示する機能は実装しない。

DiscordやXへの導線はフッターリンクとして扱う。

---

## 12. データベース利用

DBを用いたデータ保存、検索、ユーザー管理、投稿管理は初期スコープ外とする。

本サイトは静的サイトとして動作し、Markdown / MDX / Astroページ、変換済みJSON、静的アセットのみでビルド・公開できることを前提とする。

SQLite、PostgreSQL、MySQL、Firestore、Supabase等の導入は行わない。

---

## 13. サーバーサイド処理

常駐サーバー、APIサーバー、SSR、サーバーアクション、バックエンド処理は初期スコープ外とする。

検索、データ表示、ページ生成は静的サイトとして成立する範囲で実装する。

公開環境はGitHub Pages等の静的ホスティングを前提とし、Node.jsサーバーの稼働を要求しない。

---

## 14. 外部検索サービス連携

Algolia、ElasticSearch、Meilisearchなどの外部検索サービス連携は初期スコープ外とする。

検索はPagefind等を用いた静的検索を第一候補とし、公開用ビルド時に検索インデックスを生成する。

外部サービスのAPIキー、認証情報、Secretsを必要とする検索実装は行わない。

---

## 15. PDF自動生成

ルールサイトの内容からPDFルールブックを自動生成する機能は初期スコープ外とする。

初期公開ではWebサイトとしての閲覧性と更新性を優先する。

PDF化、印刷用CSS、冊子レイアウト、DTP向け出力は将来検討とする。

---

## 16. PWA対応

オフライン閲覧、ホーム画面追加、Service Worker、キャッシュ制御などのPWA対応は初期スコープ外とする。

初期実装では、通常の静的Webサイトとして軽量に閲覧できることを優先する。

PWA化に伴うキャッシュ事故や更新反映遅延を避ける。

---

## 17. 多言語対応

英語版、翻訳切り替え、多言語ルーティング、i18n対応は初期スコープ外とする。

初期公開は日本語サイトとして構築する。

将来的に多言語化する可能性は排除しないが、初期実装で翻訳管理や言語切り替え機能は作らない。

---

## 18. 高度な画像最適化

画像の自動リサイズ、複数解像度生成、CDN連携、高度なレスポンシブ画像生成は初期スコープ外とする。

初期実装では、画像を静的アセットとして管理し、Markdown / MDXまたは画像Componentから表示できることを優先する。

可能な範囲で `loading="lazy"` を設定する程度に留める。

タイトルロゴ、キービジュアル、ワールドガイド用画像などを静的アセットとして表示すること自体は初期スコープ内とする。

---

## 19. 高度な一覧フィルタ

スキル一覧・アイテム一覧に対する複雑な絞り込み、複数条件検索、ソートUI、タグ検索、保存可能なフィルタ条件は初期スコープ外とする。

初期実装では、カード表示とサイト内検索を優先する。

簡易フィルタは将来拡張候補とする。

---

## 20. 用語集専用ページ

専用の用語集ページ `/glossary` は初期スコープ外とする。

初期実装では、用語説明は `/introduction`、`/rules`、各データページ内で扱う。

固有用語が増え、参照性が不足した場合に将来追加を検討する。

---

## 21. パンくずリスト

パンくずリストは初期スコープ外とする。

現在地表示は、サイトメニューの現在ページハイライトで担保する。

スマホ閲覧時にスクロール後見えづらくなる要素を増やさない。

---

## 22. ページ内目次の現在位置ハイライト

ページ内目次で現在閲覧中の見出しをハイライトする機能は、初期必須要件には含めない。

ページ内目次自体は実装するが、スクロール位置に応じた現在見出しハイライトは優先度低とする。

実装が容易であれば対応してもよいが、初期公開を阻害してはならない。

---

## 23. 個別OGP画像生成

各ページごとに専用OGP画像を自動生成する機能は初期スコープ外とする。

OGPメタ情報は必須とするが、初期実装では共通OGP画像を利用してよい。

個別OGP画像の作成・自動生成は将来検討とする。

---

## 24. CSS と JavaScript 間のブレークポイント定数一元化

CSS の media query と JavaScript の `matchMedia` で使うブレークポイント値を、単一の定数定義から自動共有する仕組みは初期スコープ外とする。

CSS custom property は media query の条件値として使えないため、一元化には `@custom-media` のビルド時変換、または CSS と TypeScript の定義を生成する仕組みが必要になる。初期実装では、そのための設定・生成物・依存関係を追加しない。

値の重複には保守性上の難点があることを認識したうえで、現時点では導入コストに対する効果が見合わないものとして意図的に許容する。ブレークポイントを増やす、または同じ値の不整合が繰り返される段階で再検討する。

将来実装する場合は、CSS 側を `@custom-media`、JavaScript 側を共通 TypeScript 定数とし、両者を単一の入力から生成または検証する方針を候補とする。

---

## 25. 高度なアニメーション

ページ遷移アニメーション、カード出現アニメーション、スクロール連動演出などの高度なアニメーションは初期スコープ外とする。

本サイトでは、ルール本文とデータの読みやすさを優先する。

演出は最小限に留め、可読性・軽量性・保守性を優先する。

---

## 26. 過剰なUIライブラリ導入

大規模なUIコンポーネントライブラリの導入は初期スコープ外とする。

必要なUIは、Astro ComponentとCSSで個別に実装する。

アイコンや小規模ユーティリティの導入は許容するが、サイト全体を特定UIライブラリに依存させない。

---

## 27. アクセス解析

Cloudflare Web Analytics、Plausible、Umamiなどのアクセス解析は初期スコープ外とする。

初期実装では、外部サービスのスクリプト、トラッキングID、Secretsを必要とする構成を追加しない。

アクセス解析を追加する場合は、ルールサイト本体が安定した後、独立したタスクとして扱う。

---

## 28. Web上のデータ編集・変換実行

Web上でExcelをアップロードして変換する機能、ブラウザから変換スクリプトを実行する機能、変換結果をWeb上で編集する機能は初期スコープ外とする。

Excel変換は、制作者がローカルで明示的に実行するメンテナンス作業とする。

CI/CDではExcel変換を必須工程にしない。

---

## 29. `.raw/contents` のGit管理

`.raw/contents/SLUG.md` は、ユーザーがローカルで画面作成用のcontents markdownを置くための作業入力であり、Git管理しない。

`.raw/contents` の内容を公開サイトから直接参照する仕組みは初期スコープ外とする。

対応するcontentsのユーザー編集Markdown本文とHTMLコメントは、ページ本文・可視の表示構成の正本とする。`src/pages` 配下の `.mdx` または `.astro` は、その指示を反映する公開用実装とする。

---

## 30. 凡例専用Component

`SkillLegend`、`ItemLegend`、`WeaponLegend`、`ArmorLegend`、`OmamoriLegend`、`CyberneticLegend`、`NanomachineLegend`、`DrugLegend` のような凡例専用Componentは初期スコープ外とする。

スキル凡例は、`SkillCard` に凡例用データを渡して表示する。

アイテム凡例は、各Item系Cardに凡例用データを渡して表示する。

各Item系Cardは、通常のデータ表示と凡例表示を担う。

---

## 31. 汎用ItemCardへの統合

武器、防具、お守り、サイバネ、ナノマシン、ドラッグを、単一の汎用 `ItemCard` Componentへ無理に統合する方針は初期スコープ外とする。

各アイテム種別は表示項目が異なるため、初期実装では以下のように個別Componentを用意する。

- `WeaponCard`
- `WeaponList`
- `ArmorCard`
- `ArmorList`
- `OmamoriCard`
- `OmamoriList`
- `CyberneticCard`
- `CyberneticList`
- `NanomachineCard`
- `NanomachineList`
- `DrugCard`
- `DrugList`

共通ロジックや共通CSSの再利用は許容するが、カードComponent自体はアイテム種別ごとに分ける。

---

## 32. ブラウザ標準 Text Fragment による検索遷移先ハイライト

検索結果からの遷移URLへ `#:~:text=` を付与し、ブラウザ標準の Text Fragment で本文中の一致語をハイライトする機能は初期スコープ外とする。

現行の検索結果遷移では、Pagefind のハイライト機構を用いる。Text Fragment はブラウザごとの対応差、検索語と本文の一致箇所の特定、URLへの文字列エンコードを追加で考慮する必要があるため、初期公開で置き換えない。

遷移先ハイライトをブラウザ標準機能へ移行する必要が生じた場合は、個別の検索改善タスクとして、既存アンカーとの併用、未対応ブラウザでのフォールバック、`::target-text` の表示差を確認してから検討する。

## 33. スコープ内と混同しやすい項目

以下は初期スコープ外ではない。

- Astro + MDX + TypeScript による静的サイト構築
- GitHub Actionsによる基本デプロイ
- Pagefindによる静的検索
- Pagefind検索index生成込みのデプロイ
- PC左サイトメニュー
- スマホ用サイトメニュー
- PC右ページ内目次
- スマホ用ページ内目次
- サイトメニューの現在ページハイライト
- Markdown / MDX / AstroページへのJS Component埋め込み
- `ImageBlock` による画像表示
- `Callout` による補足・注意・例示表示
- `SkillCard` / `SkillList`
- 各Item系Card / List
- 流儀・生き様の共通テンプレート生成
- ExcelからJSONへのローカル変換
- 変換済みJSONのGit管理
- JSON検証スキーマ
- 変換スクリプトと検証スキーマのテスト
- `docs/conversion/*.md` による変換仕様管理
- データカード個別アンカー
- GitHub Pages等のサブパス公開対応
- 共通OGPメタ情報
- 共通OGP画像
- 404ページ
- ページ下部の前後ナビゲーション
- ヒーロー画像の表示前からの領域確保と、必要に応じたskeleton表示
- Webキャラクターシートでのキャラクター作成・編集

これらは `docs/plan.md` および `docs/requirements.md` に従って初期実装対象として扱う。
