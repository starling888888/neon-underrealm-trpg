# NPC変換仕様

## 目的

`.raw/data/npcs.xlsx`を、`/world`のNPC紹介で利用する公開用JSONへ変換する。

この仕様は、`42-0-npc-data-normalization`で実装する変換処理、schema、取得層の契約である。Excel本体はGoogle Driveを正本とするローカル作業入力であり、Git管理しない。

## 入出力

| 項目             | 値                         |
| ---------------- | -------------------------- |
| 入力Excel        | `.raw/data/npcs.xlsx`      |
| 対象シート       | `npcs`                     |
| 出力JSON         | `data/generated/npcs.json` |
| JSONの`dataName` | `npcs`                     |
| 変換コマンド     | `npm run convert:npcs`     |
| データ取得層     | `src/lib/data/npcs.ts`     |

変換は制作者がローカルで明示実行する。CI/CDと通常のsite buildは、入力Excelおよび変換コマンドに依存せず、Git管理された生成JSONだけを使う。

## 入力シート

`npcs`シートは、1行目を列見出し、2行目以降をデータ行とする。列見出しは順序を含めて完全一致とし、列追加、列削除、列移動は入力エラーとする。

| 列 | Excel見出し | JSONフィールド    | 入力規則                                    |
| -- | ----------- | ----------------- | ------------------------------------------- |
| A  | `グループ`  | `group`           | 任意の1行テキスト。空欄を許可する。         |
| B  | `ID`        | `id`              | 必須。英小文字、数字、`_`のみを使うNPC ID。 |
| C  | `名前`      | `name`            | 必須の1行テキスト。全NPCで一意。            |
| D  | `二つ名`    | `epithet.text`    | Eと対で任意の1行テキスト。                  |
| E  | `ルビ`      | `epithet.reading` | Dと対で任意の1行テキスト。                  |
| F  | `セリフ`    | `quote`           | 必須の1行テキスト。                         |
| G  | `説明`      | `description`     | 必須テキスト。改行を保持する。              |

空欄行は末尾だけを許可する。データ領域内の完全空行、部分空行、余分な非空セルは入力エラーとする。文字列の先頭・末尾の空白は除去し、説明中の改行はLFへ正規化する。

`二つ名`と`ルビ`は、両方が空欄なら`epithet: null`、両方が入力されていれば`epithet`オブジェクトへ変換する。片方だけの入力はエラーとする。

## IDと表示順

- `id`は`^[a-z][a-z0-9_]*$`に一致し、全データで一意でなければならない。
- `id`は入力Excelから受け取り、自動採番や名前からの生成をしない。
- `id`と`name`の対応は恒久的に維持する。既存の生成JSONがある再変換では、既存IDの削除、既存IDへの別名の割り当て、既存名称への別IDの割り当てをエラーにする。
- 出力配列はExcelの入力順を維持し、`sourceOrder`に1からの連番を記録する。ページ側や取得層で独自に並べ替えない。

## 出力JSON

出力は次の形とする。`updatedAt`は変換時刻をJSTオフセット`+09:00`付きISO 8601形式で記録し、データ内容が同一の再変換では既存値を維持する。

```json
{
  "dataName": "npcs",
  "updatedAt": "2026-07-22T00:00:00+09:00",
  "data": [
    {
      "id": "oogusu_arma",
      "group": "ヤクザ",
      "name": "オオグス・アルマ",
      "epithet": null,
      "quote": "話を聞こうか。貴様は何を賭ける？",
      "description": "互助組合ヨドの盟主の一つ、オオグス組の組長。69歳。",
      "sourceOrder": 1
    }
  ]
}
```

この変換は画像パスをExcelまたは生成JSONへ持ち込まない。`id`を個別画像の対応キーとして利用するか、`no_image.webp`をどの条件で使うか、公開アセットの最終パスは42-0のissueで既存SSoTと未追跡画像を照合して決める。

## 検証

`NpcSchema`は各データ行の型、必須項目、改行、二つ名とルビの組み合わせを検証する。`NpcJsonSchema`または同等のassert helperは、JSON全体について次を検証する。

- `dataName`が`npcs`であること
- `updatedAt`がJSTオフセット付きISO 8601形式であること
- `group`が空文字列を含む1行テキストであること
- IDの書式と一意性、名称の一意性
- `epithet`が`null`または完全な`text`と`reading`を持つこと
- 出力配列が空でないこと
- `sourceOrder`が1からの連番で、Excel入力順と一致すること

入力エラーには、Excelの行番号、列記号、列見出しを含める。

## テスト

実Excel本体に依存しない最小fixtureを用意し、少なくとも次を確認する。

- 7列・1行ヘッダー・任意のグループ値または空欄を正しく変換できること
- 出力順と`sourceOrder`がExcel入力順を維持すること
- ID・名称の重複、不正なID、必須項目欠落、部分空行を拒否すること
- 二つ名とルビの片方だけの入力を拒否し、両方空欄を`epithet: null`へ変換すること
- ヘッダー不一致、列追加、列削除を行・列・見出しを含むエラーで拒否すること
- 既存IDの削除・IDと名称の対応変更を拒否し、新規NPCの追加を許可すること
- CRLFをLFへ正規化し、説明中の改行を保持すること
- 取得層が全NPCの入力順配列とID指定の1件を返すこと
- 入力Excelなしで、生成済み`npcs.json`を使ったsite buildができること

## 実データ確認時点の差分

2026-07-22に同期した`npcs`シートは11件である。現行`/world`の静的`NpcCard`は9件であり、`sanjai`と`touryou`が未掲載である。この表示差分と、`touryou`用の個別画像またはfallback画像の扱いは、変換実装前に42-0のissueで対象範囲として明記する。
