# 12-1-site-menu-layout-copy — サイトメニューの文言と階層レイアウトを調整する

## 目的

PC左サイドバーおよびスマホdrawer内で使用している共通サイトメニューについて、表示文言と階層レイアウトを調整する。

現在、`SiteMenu.astro` 本体に `SITE MENU` / `サイトメニュー` の見出しが表示されているが、サイドバー本体の情報密度を下げ、メニュー項目そのものを先に見せるため、この文言はサイドバー本体から削除する。

一方、スマホdrawerを開いたときのヘッダー領域には、閉じるボタンの横に短い状態表示として `メニュー` を表示する。

## 背景

`12-mobile-menu` では、既存の `SiteMenu.astro` / `SiteMenuItem.astro` / `siteMenuItems` をPC左サイド常設表示とスマホdrawer表示の両方で再利用する方針で実装した。

この方針自体は維持する。ただし、現在の見た目には以下の調整余地がある。

- サイドバー本体に `SITE MENU` / `サイトメニュー` が表示され、メニュー項目の前に説明的な見出しが挟まっている
- スマホdrawerでは、drawerヘッダー側にも `SITE MENU` があり、さらにdrawer内の `SiteMenu.astro` 側にも `SITE MENU` / `サイトメニュー` が出るため、文言が重複する
- 子項目開閉トグルが項目左側にあり、リンクテキストの左側に余白が生まれている
- 階層が深い項目ほど、左側の空きが目立ち、メニュー全体の視線開始位置が揃いにくい

## 方針

### 1. サイドバー本体から「サイトメニュー」系の表示文言を削除する

`SiteMenu.astro` 本体から、以下の可視テキストを削除する。

- `SITE MENU`
- `サイトメニュー`

ただし、アクセシビリティ上の `nav aria-label="サイトメニュー"` は削除しない。

可視見出しを削除する理由は以下。

- 左サイドバーは見た目と配置だけでナビゲーション領域だと分かる
- メニュー項目そのものを上に詰めて表示した方が参照性が高い
- スマホdrawer内でヘッダー見出しと二重表示になることを避ける
- 後続のページ内目次と視覚階層を分けやすくする

### 2. スマホdrawerのヘッダー文言は「メニュー」にする

スマホdrawerを開いたとき、バツボタン横の表示文言は以下とする。

```text
メニュー
```

採用理由。

- 一般的なUIラベルとして短く、理解しやすい
- drawerを開いた状態の見出しとして自然
- `サイトメニュー` より硬くない
- `目次` と異なり、後続のページ内目次機能と衝突しない
- サイドバー本体から「サイトメニュー」文言を削る方針と矛盾しない

補足。

- close button の `aria-label` は `サイトメニューを閉じる` のままでよい
- drawer領域の `aria-label` も `スマホ版サイトメニュー` または `サイトメニュー` のように、スクリーンリーダー向けには用途が分かる文言を維持してよい
- 可視テキストだけを簡潔にする

### 3. 子項目開閉トグルを右端へ移動する

現在、子項目開閉トグルが項目左側にあるため、リンクテキストの左側に常にトグル用スペースが生まれている。

これを以下の構造へ変更する。

- リンクテキストは左寄せ
- 子項目を持つ項目のみ、右端に開閉トグルを表示
- 子項目を持たない項目では、左側に不要なspacerを置かない
- トグルはリンクとは別のbuttonとして維持する

目的。

- メニュー項目の開始位置を揃える
- 左側の無駄な余白を減らす
- 階層表示を維持しつつ、本文の読み始め位置を安定させる
- PC左サイドバーとスマホdrawerで同じ見た目方針を共有する

### 4. 階層インデントを見直す

トグルを右端へ移動したうえで、階層ごとのインデント量を調整する。

方針。

- level 1 は左端基準
- level 2 は控えめにインデント
- level 3 はさらに控えめにインデント
- トグル用スペースによって左側が空きすぎないようにする

過剰なツリーUI感ではなく、ルールサイトの章立てナビゲーションとして読める程度に留める。

- サイドバー本体から見出しを削っても、ナビゲーション領域として分かりにくくなっていないか
- スマホdrawerヘッダーの `メニュー` が短すぎて意味不明になっていないか
- `目次` と誤認される文言になっていないか
- 子項目トグルを右端に移動しても、開閉可能な項目だと分かるか
- トグルとリンクのタップ領域が近すぎないか
- PC左サイドバーとスマホdrawerで見た目の思想が分裂していないか
- 後続の `15-current-menu-highlight` に不要な実装負債を残していないか
- 将来、流儀・生き様リストをメニューに追加したときにも破綻しない余白設計になっているか

## 備考

可視テキストとしての推奨は `メニュー`。

候補比較。

| 候補           | 評価                                                                 |
| -------------- | -------------------------------------------------------------------- |
| メニュー       | 最有力。短く、一般的で、drawer見出しとして自然。                     |
| サイトメニュー | 正確だが硬い。サイドバー本体から文言を削る方針とはやや噛み合わない。 |
| ナビゲーション | 意味は正確だが、UI文言として長く硬い。                               |
| 目次           | 非推奨。後続のページ内目次と衝突する。                               |
| MENU           | デザイン上は締まるが、日本語サイトでは意味伝達がやや弱い。           |

このIssueでは、サイドバー本体は無題にし、スマホdrawerヘッダーだけ `メニュー` と表示する。
